# How to implement Incremental Static Regeneration (ISR)
<span style="color: #808080;">Last updated March 25, 2026</span>

## Examples
* [Next.js Commerce](https://vercel.com/templates/next.js/nextjs-commerce)
* [On-Demand ISR](https://on-demand-isr.vercel.app)
* [Next.js Forms](https://github.com/vercel/next.js/tree/canary/examples/next-forms)

증분형 정적 재생(ISR)을 통해 다음을 수행할 수 있습니다:
* 전체 사이트를 다시 구축하지 않고 정적 콘텐츠 업데이트
* 대부분의 요청에 대해 사전 렌더링된 정적 페이지를 제공하여 서버 로드를 줄입니다.
* 적절한 `cache-control` 헤더가 페이지에 자동으로 추가되는지 확인하세요.
* 긴 `next build` 시간 없이 대량의 콘텐츠 페이지를 처리합니다.

다음은 최소한의 예입니다:

app/blog/[id]/page.tsx
```tsx
interface Post {
  id: string
  title: string
  content: string
}

// Next.js는 요청이 들어올 때 최대 60초마다 캐시를 ​​무효화합니다.
export const revalidate = 60

export async function generateStaticParams() {
  const posts: Post[] = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const { id } = await params
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```

이 예제의 작동 방식은 다음과 같습니다:
1. `next build` 중에 알려진 모든 블로그 게시물이 생성됩니다.
2. 이 페이지(예: `/blog/1`)에 대한 모든 요청은 캐시되어 즉시 이루어집니다.
3. 60초가 지난 후에도 다음 요청은 여전히 ​​캐시된(현재는 오래된) 페이지를 반환합니다.
4. 캐시가 무효화되고 페이지의 새 버전이 백그라운드에서 생성되기 시작합니다.
5. 성공적으로 생성되면 다음 요청은 업데이트된 페이지를 반환하고 후속 요청을 위해 이를 캐시합니다.
6. `/blog/26`이 요청되고 존재하는 경우 페이지는 요청 시 생성됩니다. 이 동작은 다른 [dynamicParams](/docs/app/api-reference/file-conventions/route-segment-config/dynamicParams) 값을 사용하여 변경할 수 있습니다. 그러나 게시물이 존재하지 않으면 404가 반환됩니다.

## Examples
### Time-based revalidation
/blog의 블로그 게시물 목록을 가져와 표시합니다. 한 시간이 지난 후에도 다음 방문자는 빠른 응답을 위해 페이지의 캐시된(오래된) 버전을 즉시 받게 됩니다. 동시에 Next.js는 백그라운드에서 새로운 버전의 재생성을 트리거합니다. 새 버전이 성공적으로 생성되면 캐시된 버전이 대체되고 후속 방문자는 업데이트된 콘텐츠를 받게 됩니다.

app/blog/page.tsx
```tsx
interface Post {
  id: string
  title: string
  content: string
}

export const revalidate = 3600 // 매시간 무효화

export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog')
  const posts: Post[] = await data.json()
  return (
    <main>
      <h1>Blog Posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </main>
  )
}
```

재검증 시간을 높게 설정하는 것이 좋습니다. 예를 들어 1초가 아닌 1시간입니다. 더 높은 정밀도가 필요한 경우 주문형 재검증 사용을 고려해 보세요. 실시간 데이터가 필요한 경우 [동적 렌더링](/docs/app/glossary#dynamic-rendering)으로 전환하는 것을 고려해 보세요.

### On-demand revalidation with `revalidatePath`

보다 정확한 재검증 방법을 위해 `revalidatePath` 기능을 사용하여 필요에 따라 캐시된 페이지를 무효화하세요.

예를 들어 이 Server Action은 새 게시물을 추가한 후 호출됩니다. `fetch`를 사용하거나 데이터베이스에 연결하는 등 서버 컴포넌트에서 데이터를 검색하는 방법에 관계없이 이로 인해 전체 경로에 대한 캐시가 무효화됩니다. 해당 경로에 대한 다음 요청은 재생성을 트리거하고 새로운 데이터를 제공하며, 이후 후속 요청을 위해 캐시됩니다.

> **Note:** `revalidatePath`는 캐시 항목을 무효화하지만 다음 요청에서 재생성이 발생합니다. 다음 요청을 기다리지 않고 즉시 캐시 항목을 다시 생성하려면 페이지 라우터 [`res.revalidate`](/docs/pages/guides/incremental-static-regeneration#on-demand-validation-with-resrevalidate) 방법을 사용할 수 있습니다. 우리는 App Router에 즉시 재생성 기능을 제공하기 위한 새로운 방법을 추가하기 위해 노력하고 있습니다.

app/actions.ts
```ts
'use server'

import { revalidatePath } from 'next/cache'

export async function createPost() {
  // /posts 경로에 대한 캐시를 무효화합니다.
  revalidatePath('/posts')
}
```

[View a demo](https://on-demand-isr.vercel.app) and [explore the source code](https://github.com/vercel/on-demand-isr).

### On-demand revalidation with `revalidateTag`

대부분의 사용 사례에서는 전체 경로를 재검증하는 것이 좋습니다. 보다 세부적인 제어가 필요한 경우 `revalidateTag` 기능을 사용할 수 있습니다. 예를 들어 개별 `fetch` 호출에 태그를 지정할 수 있습니다:

app/blog/page.tsx
```tsx
export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog', {
    next: { tags: ['posts'] },
  })
  const posts = await data.json()
  // ...
}
```

ORM을 사용하거나 데이터베이스에 연결하는 경우 `unstable_cache`를 사용할 수 있습니다:

app/blog/page.tsx
```tsx
import { unstable_cache } from 'next/cache'
import { db, posts } from '@/lib/db'

const getCachedPosts = unstable_cache(
  async () => {
    return await db.select().from(posts)
  },
  ['posts'],
  { revalidate: 3600, tags: ['posts'] }
)

export default async function Page() {
  const posts = getCachedPosts()
  // ...
}
```

그런 다음 [Server Action](/docs/app/getting-started/mutating-data) 또는 [Route Handler](/docs/app/api-reference/file-conventions/route)에서 `revalidateTag`를 사용할 수 있습니다:

app/actions.ts
```ts
'use server'

import { revalidateTag } from 'next/cache'

export async function createPost() {
  // '게시물' 태그가 붙은 모든 데이터를 무효화합니다.
  revalidateTag('posts')
}
```

### Handling uncaught exceptions
데이터 재검증을 시도하는 동안 오류가 발생하면 마지막으로 성공적으로 생성된 데이터가 캐시에서 계속 제공됩니다. 다음 후속 요청에서 Next.js는 데이터 재검증을 다시 시도합니다. [오류 처리에 대해 자세히 알아보세요](/docs/app/getting-started/error-handling).

### Customizing the cache location
캐시된 페이지와 데이터를 내구성 있는 저장소에 유지하거나 Next.js 애플리케이션의 여러 컨테이너 또는 인스턴스에서 캐시를 공유하려는 경우 Next.js 캐시 위치를 구성할 수 있습니다. [자세히 알아보기](/docs/app/guides/self-hosting#caching-and-isr).

## Troubleshooting
### Debugging cached data in local development
`fetch` API를 사용하는 경우 추가 로깅을 추가하여 어떤 요청이 캐시되었는지 또는 캐시되지 않았는지 이해할 수 있습니다. [`로깅` 옵션에 대해 자세히 알아보세요](/docs/app/api-reference/config/next-config-js/logging).

next.config.js
```jsx
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### Verifying correct production behavior
프로덕션에서 페이지가 올바르게 캐시되고 재검증되었는지 확인하려면 `next build`를 실행한 후 `next start`를 실행하여 프로덕션 Next.js 서버를 실행하여 로컬로 테스트할 수 있습니다.

이를 통해 프로덕션 환경에서 작동하는 것처럼 ISR 동작을 테스트할 수 있습니다. 추가 디버깅을 위해 `.env` 파일에 다음 환경 변수를 추가하세요:

```bash
NEXT_PRIVATE_DEBUG_CACHE=1
```

그러면 Next.js 서버 콘솔이 ISR 캐시 적중 및 누락을 기록하게 됩니다. 출력을 검사하여 `next build` 중에 생성되는 페이지와 요청 시 경로에 액세스할 때 페이지가 어떻게 업데이트되는지 확인할 수 있습니다.

## Caveats
* ISR은 Node.js 런타임(기본값)을 사용할 때만 지원됩니다.
* [정적 내보내기](/docs/app/guides/static-exports) 생성 시 ISR은 지원되지 않습니다.
* 사전 렌더링된 경로에 `fetch` 요청이 여러 개 있고 각 요청의 `revalidate` 빈도가 서로 다른 경우 ISR에 가장 낮은 시간이 사용됩니다. 그러나 해당 재검증 빈도는 [캐시](/docs/app/getting-started/caching)에 의해 계속 유지됩니다.
* 경로에 사용된 `fetch` 요청 중 `revalidate` 시간이 `0`이거나 명시적인 `no-store`인 경우 경로가 동적으로 렌더링됩니다.
* 주문형 ISR 요청에 대해서는 프록시가 실행되지 않습니다. 즉, 프록시의 경로 재작성 또는 논리가 적용되지 않습니다. 정확한 경로를 재검증하고 있는지 확인하세요. 예를 들어 다시 작성된 `/post-1` 대신 `/post/1`입니다.
* 여러 인스턴스를 실행할 때 기본 파일 시스템 캐시는 인스턴스별로 이루어집니다. 온디맨드 재검증은 호출을 수신한 인스턴스만 무효화합니다. 공유된 [맞춤 캐시 핸들러](/docs/app/api-reference/config/next-config-js/incrementalCacheHandlerPath)를 사용하여 인스턴스 전체를 조정하세요. 전체 아키텍처는 [재검증 작동 방식](/docs/app/guides/how-revalidation-works)을 참조하세요.
* 백그라운드 재생성(재검증 중 오래된 기능)은 트리거 요청을 수신하는 인스턴스에서 실행됩니다. 요청별 청구를 사용하는 플랫폼에서는 이 백그라운드 작업이 추가 컴퓨팅으로 계산됩니다.
* `x-nextjs-cache` 응답 헤더를 사용하여 캐시 동작을 관찰할 수 있습니다. 값은 `HIT`(캐시에서 제공), `STALE`(캐시에서 제공, 백그라운드에서 재검증), `MISS`(캐시에 없음, 새로 렌더링됨) 또는 `REVALIDATED`(주문형 재검증을 통해 재생성됨)입니다.
